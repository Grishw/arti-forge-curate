# Подсистема куратора (ARTi Forge Curate) – обновлённая спецификация

## 1. Введение

Подсистема куратора — это веб-приложение, предназначенное для управления AR-контентом, поступающим из **ARTi Forge**, и его подготовки к публикации в мобильном приложении **ARTi Lens**. Основные пользователи — кураторы и организаторы мероприятий. Система обеспечивает:

- регистрацию и аутентификацию;
- создание и управление событиями и коллекциями;
- наполнение событий/коллекций работами (AR-проектами) из Forge;
- настройку отображения работ (размер, положение в пространстве);
- мониторинг статистики по событиям и работам (заглушка на MVP).

В рамках MVP реализовано **монолитное веб-приложение на Python + FastAPI** с модульной структурой. Все операции ввода-вывода (работа с файловой системой, чтение/запись данных) выполняются **асинхронно** с использованием `async/await` для обеспечения отзывчивости.

## 2. Архитектура приложения

### 2.1 Общая схема

```
+-------------------+      +-------------------+      +-------------------+
|   Frontend (SPA)  | <--> |   FastAPI App     | <--> |   JSON Storage    |
|   (NiceGUI)       |      |   (API + бизнес-  |      |   (файловая       |
+-------------------+      |    логика)        |      |   система)        |
                           +-------------------+      +-------------------+
                                   |                          
                                   v                          
                           +-------------------+      
                           |   File Storage    |      
                           |   (папки событий, |      
                           |    коллекций,     |      
                           |    архивы работ)  |      
                           +-------------------+
```

### 2.2 Схема взаимодействия с внешними системами

```
+-------------+       +------------------+       +------------------+
|  ARTi Forge | ----> |   Curator App    | <---- |  ARTi Lens       |
|  (источник) |  (1)  |   (FastAPI)      |  (2)  |  (потребитель)   |
+-------------+       +------------------+       +------------------+
                              |
                              v
                      +---------------+
                      |  Хранилище    |
                      |  (файловая    |
                      |   система)    |
                      +---------------+
```

(1) Forge отправляет работы через `POST /works/ingest`.  
(2) Lens запрашивает список событий, ищет и скачивает события через публичные API.

### 2.3 Компоненты

- **API-слой** (контроллеры) – обработка HTTP-запросов, валидация, сериализация. Используются асинхронные функции.
- **Сервисный слой** – бизнес-логика:
  - `AuthService` – регистрация, логин, JWT.
  - `EventService` – управление событиями (создание, обновление, удаление, публикация), включая создание и обновление файловой структуры события.
  - `CollectionService` – управление коллекциями, аналогично.
  - `WorkService` – приём работ от Forge, сохранение zip-архивов, чтение метаданных.
  - `AssignmentService` – добавление/удаление работ в событиях/коллекциях, настройка отображения, обновление файловой структуры.
  - `AnalyticsService` – заглушка для статистики.
  - `EventPackager` – утилита для копирования ассетов работы в папку события/коллекции и генерации `manifest.json`.
- **Слой доступа к данным** – репозитории на основе JSON-файлов (`JsonRepository`), обеспечивающие асинхронное чтение/запись через `aiofiles`. Каждая сущность хранится в отдельном JSON-файле в папке `data/`.
- **Файловое хранилище** – локальная файловая система с иерархической структурой (см. раздел 5.2).

## 3. Модели данных

### 3.1 Основные сущности

| Сущность       | Атрибуты                                                                                                 | Назначение                                           |
|----------------|----------------------------------------------------------------------------------------------------------|------------------------------------------------------|
| **User**       | id, email, hashed_password, full_name, role (curator/admin), created_at                                  | Пользователи системы (кураторы)                      |
| **Author**     | id, external_id (из Forge), name, email (опционально), created_at                                        | Автор (художник), создавший работу                   |
| **Work**       | id, external_id (из Forge), author_id, name, description, preview_path, zip_path, metadata (JSON), created_at | AR-проект, полученный от Forge                        |
| **Event**      | id, name, description, start_date, end_date, location, status (draft, published, archived), curator_id, folder_path, created_at | Мероприятие, хранит путь к своей файловой структуре |
| **Collection** | id, name, description, event_id (nullable), curator_id, folder_path, created_at | Подборка работ, также имеет файловую папку         |
| **EventWork**  | id, event_id, work_id, display_settings (JSON: {scale, position, rotation}), status (pending/approved/rejected), added_at | Связь работы с событием + настройки отображения      |
| **CollectionWork** | аналогично EventWork, но с collection_id                                                             | Связь работы с коллекцией                            |
| **Analytics**  | id, event_id (nullable), work_id (nullable), date, views, interactions, unique_users                     | Статистика (заглушка)                                |

Поле `folder_path` указывает на корневую папку, где хранятся все файлы, относящиеся к событию/коллекции.  
Поле `zip_path` в Work — путь к исходному zip-архиву, присланному Forge.

### 3.2 Детали хранения метаданных

Поле `metadata` в таблице **Work** содержит всё содержимое `project.json` и `export.json` (JSON). Это позволяет быстро получать информацию о маркерах, моделях, настройках без распаковки zip-архива при каждом запросе.

## 4. API Endpoints (REST)

Все эндпоинты возвращают JSON, аутентификация через JWT для приватных маршрутов. Все пути начинаются с `/api/v1/`.

### 4.1 Аутентификация
| Метод | Путь                   | Описание                     |
|-------|------------------------|------------------------------|
| POST  | `/auth/register`       | регистрация куратора         |
| POST  | `/auth/login`          | вход, получение JWT          |
| GET   | `/auth/me`             | информация о текущем пользователе |

### 4.2 Приём работ от Forge
| Метод | Путь                | Описание                                                                 |
|-------|---------------------|--------------------------------------------------------------------------|
| POST  | `/works/ingest`     | Принять zip-архив с работой от Forge. Требуется API-ключ в заголовке `X-Forge-Api-Key`. |

### 4.3 Работы (просмотр и управление) – приватные
| Метод | Путь                        | Описание                                       |
|-------|-----------------------------|------------------------------------------------|
| GET   | `/works`                    | список всех работ (с фильтрацией по автору)    |
| GET   | `/works/{id}`               | детали работы (включая метаданные)             |
| DELETE| `/works/{id}`               | удалить работу (только если не используется)   |

### 4.4 События – приватные (требуют JWT)
| Метод | Путь                          | Описание                                   |
|-------|-------------------------------|--------------------------------------------|
| GET   | `/events`                     | список событий текущего куратора           |
| GET   | `/events/search`              | поиск по названию события (параметр `q`)   |
| POST  | `/events`                     | создать событие (создаётся папка)          |
| GET   | `/events/{id}`                | детали события + список работ              |
| PUT   | `/events/{id}`                | обновить событие (метаданные)              |
| DELETE| `/events/{id}`                | удалить событие (удаляется папка)          |
| POST  | `/events/{id}/publish`        | опубликовать событие (статус → published)  |

### 4.5 Публичные эндпоинты для ARTi Lens (без JWT)
| Метод | Путь                          | Описание                                   |
|-------|-------------------------------|--------------------------------------------|
| GET   | `/events/public`              | список опубликованных событий              |
| GET   | `/events/public/search`       | поиск по опубликованным событиям (`q`)     |
| GET   | `/events/{id}/download`       | скачать событие в виде zip-архива (только если опубликовано) |

### 4.6 Коллекции – приватные
| Метод | Путь                          | Описание                                   |
|-------|-------------------------------|--------------------------------------------|
| GET   | `/collections`                | список коллекций текущего куратора         |
| POST  | `/collections`                | создать коллекцию (создаётся папка)        |
| GET   | `/collections/{id}`           | детали                                     |
| PUT   | `/collections/{id}`           | обновить                                   |
| DELETE| `/collections/{id}`           | удалить (удаляется папка)                  |

### 4.7 Публичное скачивание коллекций (без JWT)
| Метод | Путь                          | Описание                                   |
|-------|-------------------------------|--------------------------------------------|
| GET   | `/collections/{id}/download`  | скачать коллекцию, если она привязана к опубликованному событию |

### 4.8 Связи работ с событиями/коллекциями – приватные
Для событий:
| Метод | Путь                                    | Описание                                    |
|-------|-----------------------------------------|---------------------------------------------|
| POST  | `/events/{event_id}/works`              | добавить работу (с настройками) – обновляет папку события |
| DELETE| `/events/{event_id}/works/{work_id}`    | удалить работу из события – обновляет папку |
| PATCH | `/events/{event_id}/works/{work_id}/status` | изменить статус заявки (pending/approved/rejected) |

Аналогично для коллекций (заменить `event_id` на `collection_id`).

### 4.9 Аналитика (заглушка) – приватные
| Метод | Путь                        | Описание                           |
|-------|-----------------------------|------------------------------------|
| GET   | `/analytics/events/{event_id}` | статистика по событию              |
| GET   | `/analytics/works/{work_id}`   | статистика по работе               |

## 5. Детали реализации

### 5.1 Хранение данных (JSON вместо БД)

Вместо реляционной базы данных все сущности сохраняются в отдельных JSON-файлах в папке `data/`. Каждый файл содержит массив объектов соответствующего типа. Репозитории (`JsonRepository`) предоставляют асинхронные методы `list`, `get`, `create`, `update`, `delete`, работающие через `aiofiles`. Это упрощает начальную настройку и не требует внешних СУБД.

**Список файлов:**
- `users.json`
- `authors.json`
- `works.json`
- `events.json`
- `collections.json`
- `event_works.json`
- `collection_works.json`
- `analytics.json`

При старте приложения все файлы автоматически создаются с пустым массивом, если их нет.

### 5.2 Приём работы от Forge (`POST /works/ingest`)

- Ожидается multipart-запрос с полем `file` (zip-архив) и полями формы:
  - `external_id` – идентификатор проекта в Forge
  - `author_name` – имя автора
  - `author_email` (опционально)
  - `name` (опционально) – название работы
  - `description` (опционально)
- Защищено API-ключом, передаваемым в заголовке `X-Forge-Api-Key`.
- Система асинхронно:
  1. Сохраняет zip-архив в `storage/zips/{uuid}.zip`.
  2. Распаковывает во временную папку в отдельном потоке (`asyncio.to_thread`), читает `project.json` и `export.json`, извлекает preview-изображение.
  3. Сохраняет preview в `storage/previews/{uuid}.png`.
  4. Находит или создаёт автора по имени (и email).
  5. Создаёт запись Work с метаданными и путями.
- В ответ возвращает созданную работу.

### 5.3 Файловая структура событий и коллекций

При **создании** события (POST `/events`) система:

1. Генерирует UUID события.
2. Создаёт директорию `storage/events/{event_id}/`.
3. Сохраняет путь в поле `folder_path` сущности Event.

При **добавлении работы** в событие (POST `/events/{id}/works`):

1. Создаётся запись в `EventWork` с переданными `display_settings`.
2. Вызывается `EventPackager.add_work()`, который:
   - Распаковывает zip-архив работы (во временную папку).
   - Копирует все `.glb`-файлы в `storage/events/{event_id}/models/{work_id}_имя.glb`.
   - Копирует все изображения (кроме preview) в `storage/events/{event_id}/markers/{work_id}_имя.png` и т.д.
   - Копирует preview-изображение в `storage/events/{event_id}/previews/{work_id}_preview.png`.
   - Генерирует/перегенерирует `manifest.json` и `project.json` в корне папки события на основе метаданных всех привязанных работ.
3. При удалении работы (`DELETE /events/{id}/works/{work_id}`) вызывается `EventPackager.remove_work()`, удаляющий все файлы с префиксом `{work_id}_` и обновляющий манифест.

**Структура папки события/коллекции:**
```
storage/events/{event_id}/
├── manifest.json          # единый манифест для Lens
├── project.json           # агрегированные метаданные всех работ
├── models/
│   ├── {work_id}_model1.glb
│   └── {work_id}_model2.glb
├── markers/
│   ├── {work_id}_marker1.jpeg
│   └── {work_id}_marker2.png
└── previews/
    └── {work_id}_preview.png
```
Аналогичная структура для коллекций (`storage/collections/{collection_id}/`).

**Формат `manifest.json`**:
```json
{
  "eventId": "uuid",
  "name": "Название события",
  "description": "Описание",
  "startDate": "2026-07-01T10:00:00",
  "endDate": "2026-07-05T18:00:00",
  "location": "Москва",
  "works": [
    {
      "workId": "uuid",
      "externalId": "forge-id",
      "name": "Название работы",
      "displaySettings": { "scale": {...}, "position": {...}, "rotation": {...} },
      "projectMetadata": { /* содержимое project.json работы */ },
      "files": {
        "models": ["models/{work_id}_model.glb"],
        "markers": ["markers/{work_id}_marker.jpeg"],
        "preview": "previews/{work_id}_preview.png"
      }
    }
  ],
  "generatedAt": "2026-06-17T12:00:00Z"
}
```

### 5.4 Публичные эндпоинты для ARTi Lens

Доступны без JWT:

- `GET /events/public` – возвращает список событий со статусом `published`.
- `GET /events/public/search?q=...` – поиск по опубликованным событиям.
- `GET /events/{id}/download` – скачивание zip-архива события (только если статус `published`). Архив создаётся на лету с помощью `zipfile` и отдаётся через `StreamingResponse`.
- `GET /collections/{id}/download` – скачивание коллекции, если она привязана к опубликованному событию.

Эти эндпоинты предназначены для использования ARTi Lens для получения готовых AR-пакетов.

### 5.5 Настройка отображения

При добавлении работы куратор передаёт `display_settings`:
```json
{
  "scale": {"x": 1.0, "y": 1.0, "z": 1.0},
  "position": {"x": 0.0, "y": 0.0, "z": 0.0},
  "rotation": {"x": 0.0, "y": 0.0, "z": 0.0, "w": 1.0}
}
```
Эти значения включаются в манифест события/коллекции.

### 5.6 Аналитика

На начальном этапе аналитика является заглушкой: возвращаются нулевые значения. В будущем можно интегрировать сбор данных через веб-хуки от Lens.

### 5.7 Асинхронность

- Все операции с файловой системой выполняются с использованием `aiofiles` для чтения/записи файлов, а операции с zip-архивами вынесены в отдельные потоки через `asyncio.to_thread`, чтобы не блокировать event loop.
- Контроллеры FastAPI используют `async def` для всех эндпоинтов.

## 7. Заключение

Реализованная система полностью покрывает требования MVP:

- **Forge** самостоятельно инжектирует работы через защищённый эндпоинт `/works/ingest`.
- **Куратор** управляет событиями и коллекциями, настраивает отображение работ, публикует их (меняет статус).
- **Lens** получает список опубликованных событий и коллекций через публичные эндпоинты, ищет их по названию и скачивает готовые zip-пакеты.
- При создании события/коллекции **сразу создаётся файловая папка**, которая синхронизируется с составом работ (добавление/удаление/изменение настроек). Это обеспечивает готовый к выдаче пакет в любой момент.
- Все операции ввода-вывода выполняются **асинхронно** для максимальной производительности.
- Хранение данных в JSON-файлах упрощает развертывание и не требует внешней СУБД, при этом архитектура позволяет легко заменить JSON-репозитории на реализацию с реляционной базой данных в будущем.

**Ближайшие планы развития:**
- Интеграция веб-интерфейса куратора на NiceGUI.
